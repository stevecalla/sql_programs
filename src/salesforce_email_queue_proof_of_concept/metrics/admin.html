Data maintenance

Deletes deliberate test rows (sessions opened with ?metrics_test=1) that cost $0. Keeps any test AI call that actually spent money, so your spend record / bill reconciliation survives. Real + demo data are untouched.

System reference

• Analytics table: salesforce_email_queue_events (local MySQL; created on startup).
• Ask-your-data tables: salesforce_email_queue_ask_log, salesforce_email_queue_ask_corrections.
• Test mode: open a page with ?metrics_test=1 → events flagged is_test=1 (per URL only), purgeable under Maintenance. The Admin/Metrics links carry it; the App link is real (no flag), so normal queue work always counts. Use “App (test)” in the rail to open the app in test mode deliberately.
• No member PII is stored: counts/enums + operator staff username + queue name only.
• Admin pages reuse the app session with role admin (no second login).
• Sensitive data lives OUTSIDE the repo: auth.json, corrections.json, queue_access.json.
• Env: SF_PROD_*, OPENAI_API_KEY/ANTHROPIC_API_KEY, SF_EMAIL_QUEUE_ADMIN_*/SF_EMAIL_QUEUE_*, LOCAL_MYSQL_*.
